================================================================================
FEATURE 1: IMPROVE MAP BUILDING
================================================================================

Goal: Produce higher-quality, more accurate, and faster occupancy grid maps
      compared to the current implementation.

--------------------------------------------------------------------------------
Current Implementation Analysis
--------------------------------------------------------------------------------

Files involved:
  - ArActivity.kt  → updateSlam(), occupancy grid helpers, plane rasterisation
  - SlamEngine.kt  → pose history, Bresenham-based edge tracing, grid marking

Current approach:
  • Grid resolution: 0.20 m/cell (ArActivity.RES) and 0.25 m/cell (SlamEngine)
    — ISSUE: two different resolutions coexist; ArActivity's grid is the one
      actually sent to Flutter, while SlamEngine maintains a separate grid
      that is only used internally. This causes redundant memory usage and
      potential inconsistencies.
  • Free-space marking:
    - Camera cell + 5×5 radius (≈0.5 m) marked VISITED/FREE every frame
    - A single forward ray-cast (4 m max) marks cells as FREE
    - Horizontal planes rasterised as FREE via point-in-polygon scan
  • Wall marking:
    - Vertical planes rasterised with Bresenham edge tracing → CELL_WALL
  • Obstacle marking:
    - YOLO-detected objects get a fixed-radius footprint (per-type size)
  • No temporal filtering or confidence weighting on grid cells
  • Grid is a flat ConcurrentHashMap<GridCell, Byte> with no spatial indexing

--------------------------------------------------------------------------------
Proposed Improvements
--------------------------------------------------------------------------------

1.1  UNIFY GRID RESOLUTION
     - Remove redundant SlamEngine grid; use ArActivity's 0.20 m grid exclusively
     - Or expose resolution as a shared constant to avoid drift between the two

1.2  MULTI-RAY FREE-SPACE CASTING
     - Current: single forward ray (4 m)
     - Proposed: cast a fan of rays (e.g. 5–7 rays spanning ±30° around heading)
       to clear more free space per frame, especially in corridors

1.3  DEPTH-BASED OCCUPANCY (ARCore Depth API)
     - Current: depth mode is enabled but NOT used for grid updates
       (only for hit-testing in estimate3D)
     - Proposed: when DepthMode.AUTOMATIC is available, acquire the depth image
       each frame and project depth pixels into the grid — this gives dense
       obstacle/free info without relying solely on plane detection

1.4  PROBABILISTIC CELL UPDATES (LOG-ODDS)
     - Current: binary cell types (FREE/WALL/OBSTACLE), first observation wins
     - Proposed: use log-odds occupancy (like real SLAM systems):
       * Each observation increments/decrements a float per cell
       * Threshold to final cell type when sending to Flutter
       * Reduces noise from single bad plane detections or YOLO mis-fires

1.5  TEMPORAL DECAY FOR WALL/OBSTACLE CELLS
     - Current: once a cell is marked WALL, it stays forever (unless manually
       overwritten by FREE traversal)
     - Proposed: add a timestamp or observation count per cell; fade out walls
       that haven't been re-confirmed after N seconds → handles ARCore plane
       noise and transient objects

1.6  ADAPTIVE GRID RESOLUTION
     - Current: fixed 0.20 m everywhere
     - Proposed: use finer resolution (0.10 m) in the immediate vicinity of the
       robot for precise obstacle avoidance, and coarser (0.30 m) far away for
       performance — implemented via two grid layers or LOD tiling

1.7  PLANE BOUNDARY WALLS
     - Current: vertical planes → wall edges, horizontal planes → free interior
     - Proposed: also mark the BOUNDARY EDGES of horizontal planes as WALL/OBSTACLE
       (e.g. table edges, floor edges near stairs) — these are currently missed

1.8  INCREMENTAL MAP UPDATES TO FLUTTER
     - Current: entire ByteArray grid (w × h) is serialised every 800 ms
     - Proposed: send only changed cells (delta encoding) via a dirty-cell set
       → reduces MethodChannel payload, especially for large maps

1.9  GRID SPATIAL INDEX (QUADTREE / CHUNK MAP)
     - Current: ConcurrentHashMap with per-cell entries; bounding box recomputed
       from keys every 800 ms payload
     - Proposed: partition grid into 8×8 chunks; maintain bounding box
       incrementally → faster payload build and neighbour queries

FEATURE 2: SEMANTIC UNDERSTANDING FOR NAVIGATION
================================================================================

Goal: Enable the system to INTERNALLY understand spatial relationships among
      detected objects and use that knowledge to produce smarter, more accurate
      navigation — without exposing visual-landmark references to the blind user.

Design Principle:
      This app is for BLIND users. They cannot see visual landmarks like fire
      extinguishers, notice boards, or signs. Therefore:
        • Semantic understanding is an INTERNAL SYSTEM capability
        • It improves path quality, destination disambiguation, and context
        • Navigation instructions to the user must remain non-visual:
          distance-based, direction-based, step-count, and haptic/audio cues
        • The system uses landmarks internally (for better SLAM, loop closure,
          path cost estimation) but NEVER tells the user to "look for" or
          "turn at" a visual object

--------------------------------------------------------------------------------
Current Implementation Analysis
--------------------------------------------------------------------------------

Files involved:
  - SemanticMapManager.kt → spatial grid of SemanticObjects, merge/dedup logic
  - SemanticObject.kt     → data classes (Point3D, ObjectType, SemanticObject)
  - NavigationManager.kt  → destination selection (by type + qualifier)
  - VoiceCommandProcessor.kt → intent parsing (ObjectType + qualifier)

Current approach:
  • Objects are stored individually with type, position, confidence, observations
  • No relationships between objects are tracked
  • Navigation selects a single destination by type + qualifier (nearest/farthest
    /left_most/right_most) — no understanding of context or spatial grouping
  • No concept of rooms, zones, corridors, or functional areas
  • No co-occurrence tracking (e.g. "door near lift" is not expressible)

--------------------------------------------------------------------------------
Proposed Improvements
--------------------------------------------------------------------------------

2.1  OBJECT RELATIONSHIP GRAPH
     - Maintain a graph where nodes = SemanticObjects, edges = spatial relations
     - Edge types:
       * NEAR_TO       — distance < threshold (e.g. 2 m)
       * NEXT_TO       — adjacent, same orientation axis
       * ACROSS_FROM   — on opposite sides of a corridor/room
       * BETWEEN       — object C lies on the line segment A–B
     - Updated incrementally as objects are added/merged

2.2  ROOM / ZONE DETECTION
     - Cluster nearby objects + free-space regions into logical "rooms"
     - Use connected-component analysis on FREE cells bounded by WALL cells
     - Label rooms by their dominant object types:
       * Room with multiple CHAIRs → "seating area" / "waiting area"
       * Room with LIFT_GATE → "elevator lobby"
       * Room with WATER_PURIFIER → "pantry" / "utility area"
     - Store room boundaries as convex hulls or bounding boxes

2.3  CORRIDOR DETECTION
     - Identify narrow free-space regions (width < threshold) connecting rooms
     - Label as "corridor" / "hallway"
     - Useful for navigation instructions: "walk down the corridor"

2.4  LANDMARK-ASSISTED INTERNAL PATH PLANNING
     - Landmarks (fire extinguisher, lift gate, etc.) are used INTERNALLY to:
       * Improve loop closure and relocalization in the SLAM pipeline
       * Anchor waypoints to stable, recognisable features for better accuracy
       * Help the system disambiguate similar-looking corridors/rooms
     - User-facing instructions remain NON-VISUAL:
       * "Turn left in 3 metres" (distance-based)
       * "You have walked 10 steps, turn right" (step-count)
       * Audio/haptic cues for upcoming turns
     - The system knows a waypoint is "near the fire extinguisher" internally,
       but tells the user "Turn left in 2 metres" — NOT "Turn left at the
       fire extinguisher" (a blind person cannot see it)

2.5  SMART DESTINATION DISAMBIGUATION (SYSTEM-INTERNAL)
     - Problem: when there are multiple doors, the user currently can only say
       "nearest door" or "farthest door" — not enough to pick the right one
     - Proposed: system uses its internal semantic graph to disambiguate:
       * If user says "Take me to the door", and there are 3 doors, the system
         picks the most contextually appropriate one (e.g. the exit door from
         the current room, not a closet door)
       * System can ask clarifying NON-VISUAL questions:
         "I found 3 doors. Do you want the nearest one, or the one leading
          to the corridor?" (rooms/zones, not visual landmarks)
     - Relational queries use spatial/functional context, NOT visual references:
       * OK:  "Take me to the door on my left"
       * OK:  "Take me to the exit" (room/zone-based)
       * BAD: "Take me to the door near the fire extinguisher" (visual)
     - Extend VoiceCommandProcessor.parseIntent() for directional/functional
       qualifiers: left, right, ahead, behind, exit, entrance, corridor

2.6  OBJECT CO-OCCURRENCE MAPPING (SYSTEM-INTERNAL)
     - Track which object types frequently appear together
     - Build a co-occurrence matrix over time
     - Use INTERNALLY for predictions: if a water purifier was detected, the
       system expects a trash can nearby and can pre-plan paths accordingly
     - Helps the system suggest likely locations for objects not yet detected
     - NOT exposed as user-facing hints (user can't verify visual co-occurrence)

2.7  SEMANTIC CONTEXT FOR PATH PLANNING (SYSTEM-INTERNAL)
     - Current A* treats all FREE cells equally
     - Proposed: add cost modifiers based on semantics:
       * Prefer corridors over going through "seating areas"
       * Penalise paths that pass through narrow gaps between furniture
       * Prefer wider, obstacle-free paths (safer for blind navigation)
       * Avoid paths near hazards (stairs, drop-offs) unless that's the goal
     - All semantic reasoning is internal — user receives simple turn-by-turn
       directions with distance/step cues, not landmark references

2.8  SPATIAL AWARENESS SUMMARY (BLIND-FRIENDLY)
     - On-demand feature: user asks "What's around me?"
     - System generates a BLIND-APPROPRIATE spatial summary using:
       * Functional zones: "You are in a corridor" / "You are in a room"
       * Navigable directions: "There is a door 3 metres ahead"
       * Hazard awareness: "There are stairs 5 metres to your right"
       * Open space: "The corridor continues for about 10 metres ahead"
     - Descriptions focus on ACTIONABLE, NON-VISUAL information:
       * OK:  "There is a door 3 metres ahead on your left"
       * BAD: "To your left is a fire extinguisher" (not actionable for blind)
     - Requires: relative positioning (left/right/ahead/behind based on heading),
       room/zone knowledge, distance estimation, and hazard detection

2.9  PERSISTENT SEMANTIC MAP
     - Current: map is lost when ArActivity is destroyed
     - Proposed: serialise the semantic object graph + room/zone data to local
       storage (SQLite or JSON) so it can be reloaded on subsequent sessions
     - Enables: "Navigate to the door I found last time"
